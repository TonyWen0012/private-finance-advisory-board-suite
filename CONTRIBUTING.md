# Contributing

Thanks for helping improve this project.

## Before you open a pull request

1. Keep changes focused and easy to review.
2. Run the validation scripts when you change package structure or memory files.
3. Update both language versions when you change public-facing documentation.
4. Keep the orchestrator portable and avoid hardcoding provider-specific model names.

## Recommended checks

- `scripts/validate-package.ps1`
- `scripts/validate-suite.ps1`

## What to include in a PR

- What changed
- Why it changed
- Any impact on the suite structure, memory schema, or persona bundles
- Any follow-up work that should happen later

## Style notes

- Prefer small, reviewable commits
- Preserve the existing folder layout
- Keep documentation factual and reusable
- Avoid speculative claims about financial outcomes
